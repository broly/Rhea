## What is this?
Rhea is a personal R&D engine I've been building to explore modern GPU architecture end-to-end: explicit Vulkan, a render graph with automatic resource transitions, hardware ray tracing, and ML-based denoising trained on data generated by the engine itself.

It is not a game engine you should ship a game on. It is a working reference for how these systems fit together in ~XX k lines of C++.

## Highlights

### Hybrid rendering pipeline

Classic deferred rasterization (GBuffer, PBR, shadow mapping, SSR) combined with hardware ray-traced global illumination (VK_KHR_ray_tracing_pipeline).

<img width="2560" height="1506" alt="rtxgi_showcase_" src="https://github.com/user-attachments/assets/4e671c52-d61f-4cfe-8e9b-af68f34f30d6" />

### Neural denoiser, trained on the engine's own output

A U-Net denoiser reconstructs clean GI from a 1 spp signal + GBuffer (normals, albedo, depth). The engine doubles as the dataset generator: it exports paired 1 spp / 500 spp frames with full GBuffer for training (PyTorch).

### SVGF with explicit history invalidation

Temporal accumulation with disocclusion detection: history is validated and explicitly rejected where it breaks, trading ghosting against noise.

## RenderGraph

Passes declare their inputs/outputs; the graph derives execution order, resource lifetimes, image layout transitions and barriers automatically. Supports graphics, compute and ray tracing pipelines in one frame graph.

## Engine systems

* Bindless resources — descriptor indexing throughout the pipeline
* C++ reflection — macro-based compile-time field descriptions (pointers-to-members + names), powering serialization and tooling; no RTTI
* Multi-threaded asset loading — reflection-driven serialization, async resource streaming
* Shader permutation system — scalable variant management for GLSL / Slang
* CPU–GPU synchronization — explicit ownership and lifetime tracking for GPU resources
